# SurgeCodex Agentic Workflow

This repository defines the operating model for a multi-client software delivery workflow using Codex as the primary LLM.

The workflow is intentionally strict:

- One active ticket per client at a time.
- Multiple clients may be worked in parallel.
- Work must always be attached to the correct client subfolder.
- The latest remote code must be pulled before analysis or development.
- Work must never be pushed to `main` or `master`.
- Release activity is handled by a dedicated Release Manager bot.

## Workflow Summary

1. Product Owner validates the incoming work item.
2. Analyst identifies the code area and writes requirements.
3. Developer scopes and implements the change.
4. QA/Tester reviews the code and validates the outcome.
5. Release Manager pushes to the correct non-production branch.

If QA finds an issue, the ticket returns to Developer, then back to QA until it passes.

## Repository Layout

- `agents/` role definitions and behavioural constraints for each bot
- `workflow/` lifecycle rules, branch rules, and delivery policy
- `templates/` standard artefacts produced during execution
- `clients/` client code checkouts only
- `runtime/` workflow-owned ticket artefacts, locks, and attachments

## Intended Client Structure

Each client gets a dedicated code checkout under `clients/`.
Workflow state is stored separately under `runtime/`.

Example:

```text
clients/
  acme/
  globex/

runtime/
  acme/
    tickets/
    state/
  globex/
    tickets/
    state/
```

`clients/{slug}/` contains only the client codebase checkout.

`runtime/{slug}/tickets/` contains ticket artefacts generated by the agent chain.

`runtime/{slug}/state/active-ticket.json` is the lock file that enforces one active ticket per client.

## Default Branching Rule

- Default working branch target: `dev`
- Never push directly to `main`
- Never push directly to `master`
- If `dev` does not exist, Release Manager creates a new branch from the best available non-production base using a descriptive name

Recommended branch naming:

- `feature/<ticket-id>-<short-slug>`
- `bugfix/<ticket-id>-<short-slug>`
- `chore/<ticket-id>-<short-slug>`
- `hotfix/<ticket-id>-<short-slug>`

## Current Scope

This repo currently sets up the workflow, roles, templates, and control rules.

It now also includes a first executable queue runner for Supabase-driven ticket processing.

## Automation Entry Points

- `scripts/run_surge_queue.py` Supabase queue runner
- `scripts/git_sync_client.sh` hard-coded client repo sync guard
- `scripts/run_hourly_queue.sh` cron-friendly wrapper

## Required Runtime Configuration

Provide these via `.env.local` or environment variables:

- `NEXT_PUBLIC_SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `SURGE_PRODUCT_OWNER_USER_ID`
- `SURGE_ANALYST_USER_ID`
- `SURGE_DEVELOPER_USER_ID`
- `SURGE_QA_USER_ID`
- `SURGE_RELEASE_USER_ID`

Optional controls:

- `SURGE_QUEUE_POLL_SECONDS`
- `SURGE_MAX_PARALLEL_CLIENTS`
- `SURGE_ALLOW_HARD_RESET`
- `SURGE_DEVELOPMENT_STATUS`
- `SURGE_QA_STATUS`
- `SURGE_CODEX_SANDBOX`

## Current Limits

- Image attachments are downloaded and can be passed to Codex.
- Video attachments are downloaded and catalogued, but transcription and frame extraction are not yet automated because `ffmpeg` is not installed.
- The runner assumes client repos already exist under `clients/{slug}`.
- GitHub auth is expected to already work on the server for the client repos.
- Workflow artefacts are intentionally kept out of client repositories and live under `runtime/`.
